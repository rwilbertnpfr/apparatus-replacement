# NPFR Fleet & Equipment Replacement Dashboard — `/fleet`

Read-only web dashboard for viewing the North Port Fire Rescue apparatus and equipment replacement schedule. Companion app to `/splan`, same architecture. Intended to be hosted on GitHub Pages at:

```
https://northportfirerescue.github.io/npfr-fleet-replacement
```

---

## Repository structure

```
fleet/
├── index.html                        # Application shell and all JS logic
├── style.css                         # All styles — themes, vehicle-type colors, layout
├── build.py                          # Pre-push build script (index + Excel)
├── README.md                         # This file
├── public/
│   └── Logo.png                      # NPFR shield logo (header) — drop yours in here

└── exports/
    └── NPFR_Fleet_Replacement_YYYYMMDD.xlsx   # Generated by build.py
```

## Data model

All fleet data lives in a single JSON file under `data/`. This is the complete source of truth — `index.html` and `build.py` contain no hardcoded fleet data. Cost escalation, years-to-replacement, and totals are all computed client-side in `index.html` (and, separately, via formulas in the Excel export) from these raw inputs — same "raw data in, everything else derived" philosophy as `/splan`.

```json
{
  "exported": "2026-07-15T00:00:00",
  "label": "Initial fleet data load",
  "escalators": {
    "Engine": { "vehicle": 0.11, "equipment": 0.06 },
    "Aerial": { "vehicle": 0.11, "equipment": 0.06 },
    "Rescue": { "vehicle": 0.08, "equipment": 0.06 },
    "Staff":  { "vehicle": 0.08, "equipment": 0.06 }
  },
  "units": [
    {
      "cid": "74523",
      "unit": "E81",
      "type": "Engine",
      "inServiceYear": 2024,
      "originalCost": 743872,
      "replacementYear": 2034,
      "equipmentCost": 435000
    }
  ]
}
```

### Fields

| Field | Description |
|---|---|
| `cid` | County/agency asset ID. `"NEW"` is used for unassigned units — **not unique**, don't key lookups on it. |
| `unit` | Unit identifier (e.g. `E81`). Always unique — use this as the lookup key. |
| `type` | One of `Engine`, `Aerial`, `Rescue`, `Staff`. Drives color coding throughout. |
| `inServiceYear` / `replacementYear` | Used to compute years-to-replacement. If `replacementYear < inServiceYear`, the dashboard flags the row (data-entry error) rather than silently computing a negative-years escalation. |
| `originalCost` / `equipmentCost` | Vehicle and equipment cost at time of purchase, escalated separately using `escalators[type]`. |

## Application views

| View | Description |
|---|---|
| **Overview** | Program summary strip (unit count, current fleet value, projected replacement cost, next due year) + one card per vehicle type |
| **Timeline** | Collapsible year-by-year bands, units due grouped and totaled per year |
| **Units** | Full sortable, filterable table — filter by vehicle type or replacement year, search by unit/CID#, click any column header to sort |
| **Forecast** | Stacked bar chart of replacement cost by year (colored by vehicle type) with a cumulative-total overlay line, peak-year callout |

Read-only by design — no edit mode. To change fleet data, edit the JSON directly (see Workflow below).

## Workflow

### Updating fleet data

```bash
cd npfr-fleet-replacement
python -m http.server 8080
# Open http://localhost:8080 to preview current data
```

1. Edit (or copy and re-date) the JSON file in `data/` — e.g. `data/fleet_20260901.json`. Update `exported` and `label`.
2. Run `build.py` to regenerate `data/index.json` and the Excel export:
   ```bash
   python build.py
   ```
3. Commit and push:
   ```bash
   git add .
   git commit -m "fleet update 20260901 — description of what changed"
   git push
   ```

GitHub Pages deploys automatically within ~30 seconds. The dashboard always loads the most recently dated `fleet_*.json` file listed in `data/index.json`.

### Adding a new apparatus

Add a new object to the `units` array in the current `data/fleet_*.json` file. If it's a new vehicle type, add a matching entry to `escalators` first — the dashboard and Excel export both key off `escalators[type]`, so an unmapped type will show blank costs.

---

## build.py

Run before every `git push`. Reads the latest `fleet_*.json` from `data/` and:

1. Regenerates `data/index.json` listing all available data files
2. Builds a themed Excel workbook (flat `Data` table, one row per Vehicle/Equipment line item, native Excel Table for pivoting) saved to `exports/`
3. Writes the export filename back into the source JSON so the dashboard's **↓ Excel** button always points at the current file
4. Prints a per-vehicle-type cost summary to the terminal

```bash
python build.py
```

Requires `openpyxl`:
```bash
pip install openpyxl --break-system-packages
```

## Excel export

The **↓ Excel** button downloads the most recently built workbook from `exports/`. It does not generate on the fly — run `build.py` first. The workbook's `Data` sheet is a native Excel Table (`FleetData`) with dropdown validation and a few blank future-apparatus rows, ready to use as a PivotTable source (Insert ▸ PivotTable ▸ select `FleetData`).

---

## Local development

No build step, no bundler, no Node.js required for the dashboard itself (`build.py` needs Python + openpyxl).

```bash
cd npfr-fleet-replacement
python -m http.server 8080
# Open http://localhost:8080
```

The app fetches `data/index.json` and the latest data file on load — this requires a server (`fetch()` does not work on `file://`).
